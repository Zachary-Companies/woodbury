## Executive Summary

Between March 5 and March 12, Woodbury moved in five clear directions:

1. Browser recording became more resilient and more usable by shifting toward accessibility-derived targeting and by capturing file-picker intent explicitly.
2. The dashboard expanded from a single configuration surface into a much broader operations console: chat, assets, MCP management, training, storyboards, batch runs, social tooling, and a much more capable compositions UI.
3. The v3 agent loop matured from partial wiring into a real subsystem with planning, closure, verification, recovery, skill routing, state management, and policy surfaces.
4. Pipeline generation became more disciplined. Instead of treating generation as a one-shot act, the system now treats reusable pipelines as a staged lifecycle: design, generate, validate/repair, and verify.
5. Pipelines became more shareable and more fault-tolerant. Composition inputs can now be surfaced as form fields, and failing script nodes can inspect graph context, repair edge mismatches, and attempt targeted auto-fixes.

The net effect is that the repo shifted from feature accumulation toward stronger contracts. The March 11 and March 12 work in particular shows a pattern: large surfaces were added first, then the team started tightening invariants, documentation, and generation discipline around them.

### Add accessibility-first recording mode, file dialog capture, and toolbar cleanup

#### What changed

- The Chrome extension recording path was expanded substantially in [chrome-extension/content.js](/Users/andrewporter/Documents/GitHub/woodbury/chrome-extension/content.js), with support for an accessibility-first capture mode.
- Workflow recording and replay types were extended in [src/workflow/types.ts](/Users/andrewporter/Documents/GitHub/woodbury/src/workflow/types.ts), [src/workflow/recorder.ts](/Users/andrewporter/Documents/GitHub/woodbury/src/workflow/recorder.ts), [src/workflow/executor.ts](/Users/andrewporter/Documents/GitHub/woodbury/src/workflow/executor.ts), and [src/workflow/resolver.ts](/Users/andrewporter/Documents/GitHub/woodbury/src/workflow/resolver.ts).

- File-input interactions during recording began producing an explicit `file_dialog` step instead of being treated as an ordinary click.
- The compositions toolbar was decluttered by collapsing secondary actions behind a single menu.

#### Why these decisions make sense

- **Accessibility-first recording was a reliability decision.** CSS selectors are often too brittle for modern apps because class names are dynamic, DOM structure shifts frequently, and Shadow DOM boundaries break naive traversal. Moving to ARIA roles, accessible names, and SVG fingerprints indicates a deliberate attempt to anchor playback to user-visible semantics instead of implementation details.
- **Shadow DOM support was necessary, not optional.** The mention of `composedPath()` and auto-piercing shows the code was adapted for real-world component libraries, where events often originate inside closed or nested component trees.
- **Capturing file dialogs as a first-class step was about intent preservation.** A file upload is not just a click. It has a trigger element, a selected path, and often a variable placeholder. Modeling that explicitly makes playback automatable and parameterizable.
- **Toolbar cleanup was a usability tradeoff.** The change sacrifices always-visible affordances for lower visual clutter. That is reasonable because expert features like export, import, rename, copy, and delete are lower-frequency than run/edit actions.

#### Product impact

- Recorded workflows should be more stable across UI changes.
- Upload flows became recordable in a way that can be parameterized later.
- The dashboard became easier to scan during pipeline editing.

#### Risks introduced

- Accessibility metadata can also be unstable if sites do not maintain labels well.
- Resolver complexity increased, which means more paths to debug when replay misses an element.
- File dialog capture depends on placeholders and user-supplied values being handled consistently at run time.

## Cross-Cutting Decisions Across The Whole Period

### 1. The project is moving from ad hoc automation to contract-driven automation

The strongest pattern across these commits is the move from capability-first changes to invariant-first changes.

- March 5 added more robust selectors and explicit file-dialog modeling.
- March 7 added new execution surfaces, integrations, and UI layers.
- March 11 introduced a modular architecture and formal route boundaries.
- March 11 and March 12 then added lifecycle stages, validation tightening, and explicit docs.

This is the right sequence for a growing system. First you discover what surfaces you need. Then you lock down the contracts those surfaces depend on.

### 2. The dashboard is now the central product surface

The changes overwhelmingly favor the dashboard as the main operator experience.

- Chat moved into the dashboard.
- Assets, MCP, social, storyboard, schedules, workers, training, and batch execution all landed there.
- Compositions gained richer editing, execution, and now shareable form mode.

This is a clear product decision: the CLI still matters, but the dashboard is where multi-surface, stateful workflows are meant to be orchestrated.

### 3. Pipeline generation is being treated as an engineering process, not a single prompt

The lifecycle split, closure-engine hardening, validation docs, runbook creation, and exposed-variable guidance all point in the same direction.

The decision here is that reusable pipelines need:

- design intent,
- generated artifacts,
- repair/validation,
- verification,
- and a clean public interface.

That is a much more defensible posture than allowing one generated composition blob to count as success.

### 4. Shareability and operator usability are now first-class goals

Accessibility-first replay, form-mode pipeline execution, asset views, chat in dashboard, toolbar cleanup, home dashboard, and shareable composition links all point to the same conclusion: the product is trying to make more of its power available to users who are not necessarily graph authors or CLI power users.

### 5. The repo now depends much more heavily on documentation as infrastructure

The docs added on March 11 are not supplemental. They are part of how the system is expected to remain maintainable. That is especially important in a repo where both human developers and coding agents are likely to navigate by contracts and runbooks.

## Release Notes View

If this period were condensed into a release narrative, it would read like this:

- `1.2.1` stabilized a stronger browser recording model with accessibility targeting and file-dialog support.
- `1.2.2` broadened Woodbury into a larger operating environment with dashboard chat, MCP, assets, Tool nodes, and more complete pipeline semantics.
- The post-`1.2.2` work focused on internal architecture, agent loop maturity, documentation, lifecycle rigor, and making compositions safer and easier to share.

## Closing Assessment

The most important decision made during this period was not any single feature. It was the decision to stop letting major surfaces remain implicit.

You can see that in multiple places:

- selectors became more semantic,
- file upload intent became explicit,
- dashboard code was broken into owned modules,
- route families became isolated,
- chat and pipeline behavior got contract docs,
- pipeline generation became staged,
- repeated external inputs started getting normalized into explicit variable nodes,
- and script failure handling became graph-aware instead of purely local.

That is the pattern of a system trying to become operable at scale.
